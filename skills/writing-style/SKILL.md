# Skill: writing-style

## Purpose
The user's personal rules for prose style. Apply them to every piece of user-facing writing you produce, meaning chat replies, explanations, README and other docs, commit and PR bodies, and comments written as prose. They govern punctuation and formatting, not content or code.

## When to Use
Any time you write prose for this user, not only when asked to edit or polish text. Treat it as an always-on default for ordinary responses, the same way you would keep a style guide in mind while typing.

## The rules
1. No em-dashes. Never use the "—" or "–" character. Rewrite with a comma, a pair of parentheses, or two separate sentences. A reach for an em-dash is usually a sentence asking to be split.
2. No semicolons. Break the clause into two sentences, or join it with a comma and a word like "and", "but", or "so".
3. No bulleted or numbered lists unless the user explicitly asks for one. Default to flowing paragraphs and write what you would have bulleted as ordinary sentences. If the user says something like "list it" or "give me bullets", a list is then fine.
4. Use colons sparingly. Prefer a plain sentence over a colon that sets up an explanation or a list. A colon in front of a short quoted term, a path, or a ratio is fine. Do not use one to staple two thoughts together.

## Why
This user prefers writing that reads like continuous, spoken prose rather than punctuation-heavy or list-chopped text. Em-dashes, semicolons, stacked colons, and reflexive bullets all fragment a paragraph and read as a generic AI default. Dropping them makes the writing plainer and more human.

## Notes
These rules cover user-facing prose only. They do not touch code, file paths, config syntax, or data the user asked to see in a particular shape. Markdown headings and a conventional commit body still read as prose, so the same rules apply there. If a truly tabular answer would genuinely be clearer, ask before defaulting to a list.
